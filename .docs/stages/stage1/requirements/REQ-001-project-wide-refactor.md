## 背景与目标

- 当前项目已经形成多入口、多层级、多种渲染与数据处理路径并存的结构，需要通过系统性重构提升职责清晰度与维护效率。
- 当前确认的重构主线集中在执行面，而不是无边界地覆盖整个仓库的所有问题。
- 本轮父需求聚焦两条主线：完全去除 Astrbot 及其专用设施；将 CLI 提升为一等能力并收敛为 Web 执行核心的包装层。

## 当前口径

1. 本需求是父需求，用于承载本轮执行面重构的范围、顺序、依赖和子需求拆分。
2. 本需求不再保留 Astrbot 支持，Astrbot 相关入口、适配层、专用配置、文档、依赖、测试与其他配套设施全部进入移除范围。
3. CLI 被提升为一等能力，并且除了保留当前交互式能力外，还必须支持带参数的单次执行模式；单次执行完成后输出到 stdout 并退出。
4. CLI 不再直接承载核心执行逻辑，而是改为对 Web 服务的包装层，通过请求目标 Web 服务完成执行。
5. CLI 支持配置目标 URL，默认指向本地服务地址；当 URL 属于本地且服务未运行时，CLI 自动启动本地服务后再请求；当 URL 属于远端时，只发起请求，不尝试启动远端服务。
6. 父需求完成的判定依赖全部子需求完成；父需求自身不直接承载实现细节，而负责汇总和约束。
7. 任何结构调整都应服务于职责清晰、依赖收敛、可测试性提升与后续维护成本下降。

## 验收标准

1. `SUB-01` 到 `SUB-06` 全部完成，并各自满足对应子需求文档中的验收标准。
2. Astrbot 支持及其全部专用设施被完整移除，仓库中不再保留 Astrbot 运行链路或兼容目标。
3. Web 被收敛为统一执行核心，CLI 完成 Web 包装化并同时支持交互式和单次执行模式。
4. 本地 URL 未运行时可以自动拉起本地服务，远端 URL 不尝试启动。
5. 文档可以清楚说明重构范围、执行顺序、风险点与阶段结果。
