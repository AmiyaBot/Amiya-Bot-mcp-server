## 背景与目标

- 当前 CLI 与 MCP 侧存在重复执行逻辑，如果先做 Web 执行接口，很容易再复制第三份实现。
- 本子需求的目标是先把共享执行核心抽出来，再让 Web 和 CLI 共同复用。

## 当前口径

1. 共享执行逻辑应下沉到服务层，而不是继续放在 adapter 层。
2. Web 以后是统一执行核心，但其背后仍应复用共享服务层，而不是把业务堆到路由函数里。
3. 本子需求不负责 HTTP 接口本身。

## 当前状态视图

- 当前状态：`todo`
- 当前目标：完成现有执行逻辑盘点与共享服务抽象边界设计。
- 当前已知进展：CLI 和 MCP 已知存在独立适配逻辑，尚未统一收敛。
- 下一步动作：
  1. 盘点 CLI 与 MCP 各自的执行链路。
  2. 明确共享服务的输入输出模型。
  3. 确定适配层与服务层边界。

## 功能拆解 / 实施拆解

1. 识别 CLI 与 MCP 的重复执行逻辑。
2. 设计共享服务层的输入输出模型。
3. 调整 adapter 与 service 的边界。
4. 为后续 Web 执行接口保留稳定复用点。

## 验收标准

1. 当前执行逻辑不再在 CLI、MCP、Web 三处各写一份。
2. 共享执行服务边界清晰，适配层职责收敛。
3. 后续 Web 执行入口可以直接复用共享服务层。
