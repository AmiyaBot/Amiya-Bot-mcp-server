## 背景与目标

- 当前 CLI 与 MCP 侧存在重复执行逻辑，如果先做 Web 执行接口，很容易再复制第三份实现。
- 本子需求的目标是先把共享执行核心抽出来，再让 Web 和 CLI 共同复用。

## 当前口径

1. 干员查询与术语查询的共享执行逻辑已下沉到应用服务层，不再在 CLI/MCP 适配层内重复实现。
2. 当前共享服务落点为 `src/app/services/operator_queries.py` 与 `src/app/services/glossary_queries.py`。
3. Web 执行入口与 CLI 适配层继续复用这些服务，不在路由或命令适配层重复拼装查询逻辑。

## 当前状态视图

- 当前状态：`done`
- 当前目标：共享执行逻辑已经收敛，CLI/MCP 不再各自维护重复查询实现。
- 当前已知进展：已新增应用服务文件并完成适配层清理；`src/adapters/**` 中不再直接持有 `search_source_spec`、`build_sources`、`local_glossary` 或 `search_operator_by_name` 的散落调用。
- 下一步动作：
  1. 无。后续新增查询能力继续优先落到应用服务层。

## 功能拆解 / 实施拆解

1. 识别 CLI 与 MCP 的重复执行逻辑。
2. 设计共享服务层的输入输出模型。
3. 调整 adapter 与 service 的边界。
4. 为后续 Web 执行接口保留稳定复用点。

## 验收标准

1. 当前执行逻辑不再在 CLI、MCP、Web 三处各写一份。
2. 共享执行服务边界清晰，适配层职责收敛。
3. 后续 Web 执行入口可以直接复用共享服务层。
