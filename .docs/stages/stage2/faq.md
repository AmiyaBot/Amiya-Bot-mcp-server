# Stage2 FAQ / AI 协作规则

> 用途：沉淀 AI 高频反复追问的问题，以及跨需求可复用的默认规则。

## 文档内容范围、结构与格式约定（必读）

### 1. 本文档负责什么

本文档用于沉淀 AI 在当前 Stage 中高频反复追问的问题，以及跨需求可复用的默认规则。

本文档只负责：

1. 记录跨需求共用的默认约定。
2. 记录 AI 容易重复确认的口径、优先级、判断规则与记录方式。
3. 降低后续对话中的重复澄清成本。

本文档不负责：

1. 记录某个单独需求的特殊讨论。
2. 记录完整聊天过程。
3. 替代需求文档中的验收、测试、bug 修复记录。
4. 替代 [bugs.md](bugs.md) 记录跨需求 bug / 中断条目。

### 2. 本文档固定结构

本文档必须长期保持以下结构：

1. 文档内容范围、结构与格式约定
2. 使用规则
3. FAQ 条目
4. 新增条目模板
5. 更新日志

### 3. 本文档记录格式

每个 FAQ 条目使用以下固定格式：

- `FAQ-ID`：唯一编号，例如 `STAGE2-FAQ-001`
- `问题`：AI 反复会问的真实问题
- `默认答案`：当前默认执行口径
- `适用范围`：影响哪些需求或整个 Stage
- `何时改写`：出现什么条件时需要更新本条

## 使用规则

1. 只有“会跨需求复用”或“AI 已多次重复追问”的问题，才写入 FAQ。
2. 单需求特殊规则写回对应需求文档，不写 FAQ。
3. bug 条目、打断开发的临时问题、未归属缺陷优先写入 [bugs.md](bugs.md)，而不是 FAQ。
4. FAQ 一旦形成默认口径，后续执行优先按 FAQ 处理，除非需求文档明确覆盖。
5. 若 FAQ 与单需求文档冲突：
   - 单需求内的局部特殊口径，以单需求文档为准；
   - 跨需求通用规则，以 FAQ 为准。

## FAQ 条目

### STAGE2-FAQ-001：stage2 文档体系应该如何使用？

- `问题`：stage2 当前应该先看哪些文档？更新时先改哪里？
- `默认答案`：
  1. 先看 [README.md](README.md) 了解阶段目标与当前优先级。
  2. 再看 [requirements-index.md](requirements-index.md) 找到当前活动需求。
  3. 进入对应需求文档处理该需求的全部细节。
  4. 若形成跨需求规则，再更新本文档。
- `适用范围`：整个 stage2
- `何时改写`：文档结构再次调整时。

### STAGE2-FAQ-002：stage1 的哪些结论继续有效？

- `问题`：进入 stage2 后，stage1 的哪些架构结论仍然默认有效？
- `默认答案`：stage1 已冻结的执行面口径继续有效，包括：Astrbot 不再保留；Web 仍是统一执行核心；CLI 仍是 Web 包装层；本地 URL 自动探活与后台启动、远端 URL 只请求不启动；对外安装与命令名继续统一使用 `amiyabot-cli`。
- `适用范围`：整个 stage2
- `何时改写`：只有当用户明确要求改动执行面架构时。

### STAGE2-FAQ-003：新增能力是否还能写回 stage1？

- `问题`：当前新增功能或体验增强，是否还能继续写回 stage1 需求？
- `默认答案`：不能。stage1 已完成，后续新增能力必须登记到 stage2 或更新阶段，不再混写回 stage1 父需求。
- `适用范围`：整个 stage2
- `何时改写`：只有在用户明确要求重新打开 stage1 收尾缺陷时。

### STAGE2-FAQ-004：开发过程中突然冒出的 bug 记在哪里？

- `问题`：开发某个需求时突然发现别的需求 bug，或者发现一个暂时说不清归属的问题，应该记在哪？
- `默认答案`：先记入 [bugs.md](bugs.md)。如果后续确认它明显属于某个需求，再在对应需求文档补充影响与处理结果，并在 bug 条目里回链。
- `适用范围`：整个 stage2
- `何时改写`：bug 管理策略改变时。

### STAGE2-FAQ-005：需求文档按时间线写还是按状态写？

- `问题`：单需求文档应该以时间线为主，还是以当前状态为主？
- `默认答案`：采用“状态视图 + 时间线附录”的混合结构。文档前半部分让 AI 快速接手当前状态，后半部分保留工作时间线，记录中断、恢复、测试往返与需求拉扯。
- `适用范围`：整个 stage2
- `何时改写`：需求文档模板变化时。
