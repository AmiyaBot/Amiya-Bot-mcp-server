## 背景与目标

- 当前干员查询链路虽然已经能生成详情卡片图片，但 CLI 文本输出仍然是旧版概要文本，MCP 返回的也主要是文本加图片链接，而不是语义化的字段级结构。
- 用户提出新的输出方向：CLI 增加 `--json`，加上时输出与 MCP 对齐的结构化 JSON；不加时 CLI 直接输出 Markdown 文本。
- 本需求的目标不是继续扩展旧摘要模板，而是建立一份统一的、可展示的结构化结果契约，让 CLI 与 MCP 共享同一份语义化结果数据。

## 当前口径

1. 本需求先只覆盖 `op` 命令，不默认扩大到 `skill`、`glossary` 或其他命令。
2. Service 层需要产出一份“展示用结构化 payload”，而不是直接暴露 raw domain model。
3. CLI 单次命令模式新增全局 `--json`；加上时输出结构化 JSON，不加时默认将同一份 payload 渲染为 Markdown 文本。
4. MCP 干员基础信息工具返回同一份结构化 payload，并继续附带图片 URL 或本地图片路径。
5. JSON 字段使用中文名词替换后的语义化字段，但结构优先保持嵌套，不采用大面积扁平拼接键名，例如优先使用“攻击力: { 当前, 满信赖, 增量 }”而不是“攻击力(满信赖)”这类展示型 key。
6. 现有 `operator_info.json.j2` 继续作为 PNG 渲染配置模板使用，不承担语义 JSON 输出职责；语义结果契约应落在 service / presenter 层。

## 功能拆解 / 实施拆解

1. 结果契约层：定义统一的干员查询展示 payload，覆盖当前已验证的详情字段。
2. CLI 输出层：为单次命令模式增加 `--json`，并基于同一 payload 提供 Markdown 默认输出。
3. MCP 输出层：将 `get_operator_basic` 调整为返回同一份结构化 payload，而不是仅返回文本字段。
4. 边界治理：避免与现有 PNG 渲染配置模板冲突，明确结果契约不走 card template 的 json 配置链路。

## 验收标准

1. `amiyabot-cli --json op 银灰` 能输出语义化 JSON，且结果不直接暴露 raw model。
2. `amiyabot-cli op 银灰` 能基于同一份 payload 输出 Markdown 文本，而不是继续依赖旧版概要文本。
3. MCP 干员基础信息工具返回同一份结构化 payload，并继续携带图片 URL 或图片路径。
4. 当前 PNG 渲染配置模板仍保持原职责，不与语义 JSON 输出冲突。
