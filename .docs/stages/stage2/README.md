# Stage2 文档 / README / Project Overview

> 用途：stage2 是当前项目的功能增强阶段，承接干员详情展示重构与档案数据补全相关工作。

## 文档内容范围、结构与格式约定（必读）

### 1. 本文档负责什么

本文件同时承担 `README` 与 `project-overview` 两个职责，是 stage2 的唯一总入口。

本文档只负责：

1. 说明当前 Stage 的阶段目标、范围、非目标与优先级。
2. 规定当前 Stage 文档体系的用途、边界、更新规则与记录格式。
3. 提供所有需求文档、FAQ 文档、索引文档、全局 bug 文档的导航入口。
4. 提供项目级进度视图，帮助 AI 和人类快速判断当前应该做什么。

本文档不负责：

1. 记录某个具体需求的完整拉扯过程。
2. 记录逐轮聊天原文。
3. 记录细粒度测试步骤、bug 修复细节或验收争议的全部上下文。

### 2. 本文档固定结构

本文档必须长期保持以下结构：

1. 文档内容范围、结构与格式约定
2. Stage 背景
3. Stage 总目标
4. 范围与非目标
5. 需求优先级与阶段进度
6. 文档体系说明
7. 更新规则
8. 当前阶段总览

### 3. 本文档记录格式

1. 项目级结论优先使用短句与编号列表。
2. 需求状态统一使用：`todo`、`in-progress`、`blocked`、`done`。
3. 涉及具体需求时，必须引用对应需求文档，而不是在本文档展开细节。
4. 涉及 AI 协作规则时，先写入 [faq.md](faq.md)，再在本文档中只保留摘要性入口。
5. 涉及每轮对话沉淀时，只记录“形成的决策/变更/问题”，不记录原始聊天流水。

- 当前阶段状态：in-progress
- 启动日期：2026-04-13

## Stage 总目标

1. 完成干员详情展示层的重构，提升详情卡片的信息组织和呈现效果。
2. 补齐干员详情所需的数据链路，覆盖当前已出现的关键资料字段与衍生信息。
3. 明确展示与数据补全的验收边界，避免本阶段范围继续漂移。

## 范围与非目标

### 范围

1. 干员详情相关模板、渲染视口配置与展示布局调整。
2. 干员详情相关的数据模型、仓储装载与领域组装逻辑。
3. 当前已出现的资料补全项，包括基础档案、画师、真名、潜能列表、基建技能、`favorKeyFrames`。

### 非目标

1. 不回退、不重做 stage1 已冻结的执行架构、入口策略与安装口径。
2. 不扩展到与干员详情无关的全站页面改版或泛化百科系统升级。
3. 不把尚未明确确认的其他资料补全需求默认并入本阶段。

## 需求优先级与阶段进度

### 当前优先级

1. 对当前未提交改动补齐多样本回归验证，确认详情卡片与数据链路在不同干员上稳定可用。
2. 明确图片卡片链路与文本摘要链路各自的当前完成度，不把未验证部分误写为已完成。
3. 在验证结果稳定后回写需求状态、测试记录与剩余后续项。

### 当前里程碑视图

- M1：已完成。完成 stage2 初始化、首个需求登记与验收口径确认。
- M2：进行中。当前代码已落下详情卡片重构与数据补全主干，并完成银灰样例验证。

### 单需求文档的职责

每个需求文档都用于承载该需求上的完整“结构化记忆”，并采用“状态视图 + 时间线附录”的混合结构，包括：

1. 当前目标、当前口径、当前状态；
2. 当前方案、验收标准、下一步动作；
3. 需求内 bug / 修复记录；
4. 与本需求相关的中断记录；
5. 历次变更与拉扯结论时间线；
6. 测试过程与结果；
7. 待确认问题。

### 全局 bug 文档的职责

[bugs.md](bugs.md) 用于承载不适合直接归入单个需求文档的问题，典型包括：

1. 开发 A 需求过程中暴露出的 B 需求 bug。
2. 暂时无法确认归属的历史遗留问题。
3. 跨需求共用链路上的缺陷。
4. 会打断当前开发节奏的中断事件。

## 更新规则

1. 每次和 AI 讨论后，不要求保留对话原文；只沉淀有效结果。
2. 若结果只影响单个需求，更新对应需求文档。
3. 若结果会影响多个需求或形成默认规则，更新 [faq.md](faq.md)。
4. 若结果是跨需求 bug、临时未归属 bug 或中断事件，先更新 [bugs.md](bugs.md)，再补充相关需求文档。
5. 若结果会改变总体优先级、阶段目标或完成状态，更新 [README.md](README.md) 与 [requirements-index.md](requirements-index.md)。
6. 若某个需求完成验收，必须同步更新：
   - 对应需求文档；
   - 若涉及全局 bug，更新 [bugs.md](bugs.md)；
   - [requirements-index.md](requirements-index.md)；
   - 后续需要时，再补发版摘要。

1. stage1 已完成，不再把新增能力回写到 stage1 父需求中。
2. stage2 默认只承接增量变化，不重复 stage1 已冻结的旧决策。
3. Web/CLI 执行面架构、自动探活与安装口径继续保持稳定，不作为本阶段改动对象。

6. 图片卡片链路与文本摘要链路的验证状态必须分开记录；当前只将已验证的图片卡片结果计入已落地进展。
